MODULE 3: Software Requirements

A Comprehensive Guide to Requirements Engineering

Introduction to Software Requirements

Software requirements form the foundation of any successful software project. They describe what the system should do and under what constraints it should operate. Think of requirements as a blueprint for a building‚Äîjust as architects need detailed plans before construction begins, software developers need clear requirements before writing code. Without well-defined requirements, projects often suffer from scope creep, budget overruns, and ultimately deliver products that don't meet user needs.


Requirements engineering is the systematic process of discovering, documenting, analyzing, and maintaining these requirements throughout the software development lifecycle. It bridges the gap between what stakeholders need and what developers build.

1. Functional and Non-Functional Requirements

Understanding Functional Requirements

Functional requirements describe the specific behaviors, functions, or capabilities that a system must provide. They answer the question: "What should the system do?" These are the features and operations that users directly interact with and depend upon. Each functional requirement typically describes an action the system must perform in response to an input or condition.


For instance, in an online banking system, a functional requirement might state that "the system shall allow users to transfer money between accounts." This is concrete, testable, and describes a specific capability the system must have.

Examples of Functional Requirements:

E-commerce System: The system shall allow customers to add items to a shopping cart, apply discount codes, and complete purchases using multiple payment methods.

Library Management: The system shall enable librarians to check out books, calculate late fees automatically, and send overdue notifications to borrowers.

Hospital System: The system shall allow doctors to view patient medical history, prescribe medications, and schedule follow-up appointments.

Understanding Non-Functional Requirements

Non-functional requirements (NFRs) specify how well the system should perform its functions. They define quality attributes, constraints, and standards that the system must meet. While functional requirements tell us what the system does, non-functional requirements tell us how well it does it. These requirements often determine the overall user experience and system success, even though they're not directly visible features.


Non-functional requirements are crucial because a system might have all the right features but still fail if it's too slow, unreliable, or difficult to use. They address concerns like performance, security, usability, reliability, and scalability.

2. User Requirements

User requirements are statements written in natural language, accompanied by diagrams, that describe what services the system is expected to provide and the constraints under which it must operate. These requirements are written from the user's perspective and should be understandable by people without technical training. The goal is to capture what users need to accomplish with the system, not how the system will be implemented.


Think of user requirements as the "wish list" from the people who will actually use the system. A teacher using an online grading system might express a user requirement as: "I need to be able to enter grades for all my students and have the system automatically calculate their final scores based on the weighting I've set up." Notice how this doesn't mention databases, algorithms, or user interfaces‚Äîit simply states what the user needs to accomplish.

üìù Natural Language

Written in everyday language that stakeholders can understand without technical knowledge. Avoids jargon and technical terminology.

üë• User-Centric

Focuses on user goals, tasks, and needs rather than system implementation. Describes what users want to achieve.

üìä Supported by Diagrams

Often includes use case diagrams, workflow diagrams, or mockups to visually represent user interactions with the system.

üéØ Goal-Oriented

Expresses the outcomes users want to achieve, allowing flexibility in how the system meets those needs.

Key Principle: User requirements should be written at a level that allows users to verify that the system will meet their needs without requiring them to understand technical implementation details. If a user can't understand the requirement, it's probably too technical.
Example User Requirements for a Hospital Management System:

UR1: As a doctor, I need to access a patient's complete medical history, including previous diagnoses, medications, and test results, so I can make informed treatment decisions.

UR2: As a nurse, I need to record vital signs (temperature, blood pressure, heart rate) for my patients and have the system alert me if any values are outside normal ranges.

UR3: As a patient, I want to schedule appointments online, receive confirmation messages, and get reminders before my appointment time.

UR4: As an administrator, I need to generate reports showing department-wise patient statistics, revenue, and resource utilization.

3. System Requirements

System requirements are more detailed, technical specifications that describe exactly what the system must do to satisfy the user requirements. These are written for developers, testers, and other technical stakeholders. While user requirements describe user-level services, system requirements break these down into precise technical specifications that can be implemented and tested.


System requirements serve as the contract between the client and the development team. They provide enough detail that developers know exactly what to build, and testers know exactly what to verify. Continuing our teacher example, the user requirement about entering grades would expand into multiple system requirements specifying the data structures for grades, validation rules for grade entries, algorithms for calculating weighted averages, and error handling procedures.

Types of System Requirements

Functional System Requirements

Detailed specifications of system functions, including inputs, processing, outputs, and error handling. These are precise and testable.


Example: The system shall validate that the entered grade is a numeric value between 0 and 100. If an invalid value is entered, the system shall display an error message and prevent submission.

Non-Functional System Requirements

Technical constraints and quality attributes that define system properties like performance metrics, security standards, and interface requirements.


Example: The system shall encrypt all grade data using TLS 1.3 during transmission and AES-256 when stored in the database.

Transforming User Requirements into System Requirements:

User Requirement: "I need to search for books in the library catalog."


Corresponding System Requirements:

SR1: The system shall provide a search interface accepting keywords, author names, titles, ISBN numbers, and subject categories as search criteria.

SR2: The system shall perform full-text searches across all catalog entries and return results within 2 seconds for 95% of queries.

SR3: The system shall display search results in a paginated list showing title, author, publication year, availability status, and location for each item.

SR4: The system shall support Boolean operators (AND, OR, NOT) and phrase searches using quotation marks.

SR5: The system shall rank search results by relevance using a weighted algorithm prioritizing exact title matches, then author matches, then keyword matches.

Important Distinction: User requirements express WHAT users need to accomplish, while system requirements specify HOW the system will enable users to accomplish those goals. System requirements are more detailed, technical, and serve as the blueprint for development.

4. Interface Specification

Interface specifications define how different components of a system interact with each other and with external systems. Just as electrical plugs need to match their sockets, software components need clearly defined interfaces to communicate effectively. These specifications document the communication protocols, data formats, function calls, and interaction patterns that enable different parts of the system to work together seamlessly.


There are three main types of interfaces that need specification: user interfaces (how people interact with the system), software interfaces (how software components communicate with each other), and hardware interfaces (how software interacts with physical devices). Each type requires careful documentation to ensure smooth integration and functionality.

Types of Interface Specifications

üñ•Ô∏è User Interface (UI)

Defines how users interact with the system, including screen layouts, navigation flows, input methods, and visual design standards.


Specifies: Screen designs, button placements, menu structures, input validation messages, accessibility features, and responsive design requirements.

‚öôÔ∏è Software Interface

Describes communication between software components, including APIs, data exchange formats, and integration protocols.


Specifies: API endpoints, data formats (JSON, XML), authentication methods, error codes, and integration protocols (REST, SOAP).

üîå Hardware Interface

Details how software communicates with physical devices such as printers, scanners, sensors, or other peripherals.


Specifies: Device drivers, communication protocols, data transmission rates, signal formats, and error handling for hardware failures.

üì° Communication Interface

Defines network communication protocols, message formats, and data exchange mechanisms between distributed systems.


Specifies: Network protocols (HTTP, TCP/IP), message queuing systems, data serialization, encryption standards, and timeout handling.

Why Interface Specifications Matter: Clear interface specifications prevent integration problems, reduce development time, enable parallel development by different teams, and make it easier to replace or upgrade components without affecting the entire system. They serve as contracts between different parts of the system.

5. The Software Requirements Specification (SRS) Document

The Software Requirements Specification (SRS) document is the comprehensive written record that captures all requirements for a software system. Think of it as the master blueprint that guides the entire development process. It serves multiple audiences: clients use it to verify that developers understand their needs, developers use it as their implementation guide, testers use it to create test cases, and project managers use it to plan resources and schedules.


A well-written SRS document is complete, consistent, unambiguous, verifiable, modifiable, and traceable. It bridges the gap between what stakeholders envision and what developers build, serving as the single source of truth throughout the project lifecycle. Without a solid SRS, projects often suffer from miscommunication, scope creep, and failed expectations.

6. Requirements Engineering Process

Requirements engineering is not a one-time activity but a continuous process that spans the entire software development lifecycle. It's like detective work combined with translation services‚Äîyou must discover what stakeholders really need (which they may not articulate clearly), analyze and organize this information, document it precisely, and then continually verify that the documented requirements remain accurate and complete as the project evolves.


This process involves several interconnected activities: elicitation (gathering requirements), analysis (refining and organizing them), specification (documenting them formally), and validation (confirming they're correct). Each activity feeds into the others, and the process often requires multiple iterations as understanding deepens and circumstances change.

Requirements Elicitation

The process of discovering requirements through interaction with stakeholders. This involves interviews, surveys, observation, workshops, prototyping, and document analysis. The challenge is that stakeholders often don't know exactly what they want or can't articulate it clearly.

Requirements Analysis

Examining gathered requirements to detect conflicts, ambiguities, and incompleteness. This includes prioritization, feasibility assessment, and modeling of requirements using diagrams and prototypes. The goal is to refine raw requirements into clear, consistent specifications.

Requirements Specification

Documenting requirements in a formal, structured manner that serves as an agreement between stakeholders and developers. This typically results in the SRS document and supporting materials like models and prototypes.

Requirements Validation

Ensuring that documented requirements accurately reflect stakeholder needs and are feasible to implement. This involves reviews, inspections, prototyping, and test case development to verify requirements before development begins.

Requirements Management

Tracking requirements throughout the project, managing changes, maintaining traceability from requirements through design and implementation, and ensuring that evolving requirements are properly assessed and controlled.

7. Feasibility Studies

Before committing significant resources to a software project, organizations conduct feasibility studies to determine whether the project is worth pursuing. Think of a feasibility study as a reality check‚Äîit answers the fundamental question: "Should we actually build this system?" This analysis examines whether the proposed system is technically achievable, economically justifiable, legally compliant, operationally practical, and schedulable within reasonable timeframes.


Feasibility studies save organizations from costly mistakes by identifying potential problems before major investments are made. They provide decision-makers with the information needed to approve, modify, or reject a project proposal. A thorough feasibility study examines multiple dimensions of the proposed system and delivers recommendations based on objective analysis.

Types of Feasibility Analysis

üíª Technical Feasibility

Question: Can we build it with current technology?


Examines: Available technology, required expertise, technical risks, system integration capabilities, and whether the required performance can be achieved.


Example: "Do we have developers with the necessary skills in machine learning to build this AI-powered recommendation system?"

üí∞ Economic Feasibility

Question: Is it financially worthwhile?


Examines: Development costs, operational costs, expected benefits, return on investment (ROI), payback period, and cost-benefit analysis.


Example: "Will the $2 million development cost be justified by the projected $5 million in cost savings over five years?"

‚öñÔ∏è Legal Feasibility

Question: Are there legal issues or compliance requirements?


Examines: Copyright issues, data privacy laws (GDPR, HIPAA), contractual obligations, licensing requirements, and regulatory compliance.


Example: "Does our patient management system comply with HIPAA regulations for protecting health information?"

‚öôÔ∏è Operational Feasibility

Question: Will it work in the organization?


Examines: User acceptance, organizational culture, change management needs, training requirements, and impact on existing operations.


Example: "Will employees adapt to the new automated workflow system, or will they resist the change?"

üìÖ Schedule Feasibility

Question: Can we complete it on time?


Examines: Project timeline, resource availability, critical deadlines, dependency on external factors, and risk of delays.


Example: "Can we launch the e-commerce platform before the holiday shopping season begins?"

8. Requirements Elicitation and Analysis

Requirements elicitation is often the most challenging part of requirements engineering because it involves extracting knowledge from people who may not be able to clearly articulate what they need. It's like being a detective and a translator simultaneously‚Äîyou must ask the right questions, observe how work is actually done (not just how people describe it), and translate tacit knowledge into explicit requirements.


The challenge stems from several factors: stakeholders often don't know what they want until they see what they don't want, different stakeholders have conflicting needs, people have difficulty describing their work processes in detail, and organizational politics can obscure the real requirements. Successful elicitation requires using multiple techniques and engaging stakeholders throughout the process.

Requirements Elicitation Techniques

üé§ Interviews

One-on-one or group discussions with stakeholders to gather detailed information about their needs, workflows, and pain points.


Best for: In-depth understanding of complex processes and stakeholder perspectives.

Types: Structured (fixed questions), Semi-structured (flexible agenda), Unstructured (exploratory)

üìã Questionnaires & Surveys

Written questions distributed to a large group of stakeholders to gather information efficiently from many people.


Best for: Collecting quantitative data and reaching geographically distributed stakeholders.

Advantage: Can reach many people quickly; statistical analysis possible.

üëÅÔ∏è Observation & Ethnography

Watching users perform their tasks in their natural environment to understand actual workflows and unspoken needs.


Best for: Discovering requirements that users take for granted or can't articulate.

Key insight: What people say they do often differs from what they actually do.

ü§ù Workshops & Focus Groups

Facilitated group sessions bringing together diverse stakeholders to discuss requirements, resolve conflicts, and build consensus.


Best for: Rapid requirements gathering and conflict resolution among stakeholders.

Technique: Joint Application Development (JAD) sessions are popular.

üé® Prototyping

Creating mock-ups or working models of the system to help stakeholders visualize and refine their requirements.


Best for: Clarifying user interface requirements and validating understanding.

Types: Paper prototypes, wireframes, clickable mockups, working prototypes.

üìö Document Analysis

Reviewing existing documentation, forms, reports, and procedures to understand current processes and information needs.


Best for: Understanding existing systems and regulatory requirements.

Sources: Policy documents, user manuals, forms, regulations, existing system documentation.

Requirements Analysis Techniques

Once requirements are gathered, they must be analyzed to ensure they're complete, consistent, and feasible. Analysis involves several key activities that transform raw requirements into a refined, organized set of specifications ready for implementation.

Critical Success Factor: Requirements elicitation is most successful when it's an iterative, collaborative process. Don't expect to get all requirements right in the first pass. Plan for multiple rounds of elicitation, analysis, and validation, and maintain continuous communication with stakeholders throughout the project.
Common Requirements Elicitation Challenges and Solutions:

Challenge: Stakeholders say "I'll know it when I see it" but can't describe requirements.

Solution: Use prototyping to give stakeholders something concrete to react to and refine.


Challenge: Different stakeholders have conflicting requirements.

Solution: Facilitate workshops to build consensus and use prioritization techniques to make trade-offs explicit.


Challenge: Stakeholders are too busy to participate in requirements activities.

Solution: Use efficient techniques like surveys for initial gathering, followed by focused workshops for critical decisions.


Challenge: Requirements keep changing throughout the project.

Solution: Implement a formal change management process and use agile approaches that embrace change.

Summary and Key Takeaways

Requirements engineering forms the foundation of successful software development. By thoroughly understanding and documenting what the system should do (functional requirements) and how well it should perform (non-functional requirements), we create a clear roadmap for development. User requirements capture what stakeholders need in their own language, while system requirements translate these needs into technical specifications that developers can implement.


The SRS document serves as the contract between stakeholders and developers, providing a single source of truth throughout the project. Interface specifications ensure that different components can communicate effectively. Feasibility studies prevent costly mistakes by validating that projects are technically achievable, economically viable, and operationally practical before major resources are committed.


Requirements elicitation and analysis are iterative processes that require multiple techniques and continuous stakeholder engagement. Success comes from using appropriate elicitation methods, carefully analyzing gathered information for completeness and consistency, and maintaining clear communication throughout the project lifecycle. Remember: the quality of your requirements directly impacts the quality of your final software product. Invest time in getting requirements right, and you'll save exponentially more time in development and maintenance.

Final Thought: Good requirements engineering is not about creating perfect documentation‚Äîit's about building shared understanding between all stakeholders. The goal is ensuring that everyone involved in the project has a clear, common vision of what needs to be built and why it matters.
